01 — Contexte

Présentation du projet

Dans le cadre de la communication digitale de la marque Olevia, un concours Instagram a été mis en place afin de gagner en e-réputation et d'accroître l'engagement de la communauté. L'objectif technique du stage était de concevoir un système automatisé d'analyse des participations, vérifiant l'éligibilité de chaque concurrent selon trois critères précis.

Le projet a été réalisé en binôme via VS Code Live Share : Nolan assurait la conception et la gestion de la base de données MySQL, tandis que le développement du script d'analyse Python était mené conjointement.

04 — Développement

Analyse des trois critères

// Analyse des abonnés

Première difficulté majeure : distinguer les abonnés réels des comptes suggérés par Instagram. Plusieurs stratégies ont été testées (filtrage par mots-clés "Suivi(e)", "Suivre", attribut tabindex="0"...) avant de trouver un critère fiable.

⚠ Problème
Instagram génère son contenu dynamiquement via JavaScript. Le HTML statique ne contient pas de structure claire liant un pseudo à son statut d'abonné.
✓ Solution retenue
Détection du bouton "Supprimer" (présent uniquement chez les abonnés réels, contre "Suivre" pour les suggestions). Critère stable et fiable.

// Analyse des likes

La liste complète des comptes ayant liké une publication n'est pas présente dans le HTML statique (chargement dynamique). Une approche alternative a été développée.

✓ Approche adoptée
Croisement des pseudos abonnés déjà identifiés avec le contenu HTML du post : si le pseudo est détecté dans la page de la publication, il est considéré comme ayant liké. Résultat stocké en booléen.

// Analyse des commentaires & mentions

Vérification qu'un commentaire contient au moins 3 mentions @ distinctes, avec règles de validation strictes.

// Tirage au sort final

Une requête SQL avec INNER JOIN sur les trois tables croise les critères. Un script Python sélectionne ensuite aléatoirement (ORDER BY RAND()) le gagnant parmi les participants éligibles et affiche son pseudo ainsi que ses trois recommandations.

05 — Limites & décisions

Contraintes techniques rencontrées

Instagram repose sur un rendu dynamique côté client (JavaScript). Il est donc impossible d'extraire de manière exhaustive et fiable toutes les données (likes, abonnés complets) depuis un simple HTML statique. Cette limitation ne relève pas d'une erreur de programmation, mais d'une contrainte structurelle de la plateforme.

⚠ Décision prise
Pour respecter le cadre légal, technique et pédagogique, le scraping direct d'Instagram en conditions réelles n'a pas été poursuivi. Le projet repose sur un HTML simulé et contrôlé représentant fidèlement la structure logique attendue, tout en démontrant la méthodologie et les compétences techniques mobilisées.

Outil Playwright également étudié comme alternative pour le rendu dynamique (ouverture d'un vrai navigateur + exécution JavaScript), mais non déployé en production pour les mêmes raisons.
